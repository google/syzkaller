# Updating Syzkaller Kernel Configurations Guidelines

These guidelines describe the process for updating the `syzkaller` dashboard configuration files to support new Linux kernel versions.

## Objective
Successfully update configuration fragments (`.yml`) and generate corresponding `.config` files for target kernel versions (e.g., `linux-upstream` and `linux-next`).

## Principles
- **Maximize Coverage**: We want to maximize the coverage of the new code and maintain the coverage of the old code.
- **Enable New Options**: If new options are introduced with the kernel version, we want them to be enabled if they help to shift left the bug detection or directly help to detect bugs. If they help with fuzzing but were not enabled, explain why in the commit message. These changes should be grouped into a commit titled `dashboard/config/linux: enable more configs`.
- **Reasoned Disabling**: If some options are not available anymore, disable them adding a comment in the YAML file referencing the kernel commit that removed, renamed, or moved them. Prefer formats like: `# Removed in <commit_hash> "<commit_title>"`, `# Renamed to <NEW_CONFIG> in <commit_hash> "<commit_title>"`, or `# Migrated to <NEW_LOCATION> in <commit_hash> "<commit_title>"`. If the specific commit cannot be found after a reasonable search, use the format: `# Removed in vX.Y (exact commit not found)`. If the option is not mentioned in any generated config files (i.e., it was not enabled in any instance), you can remove it from the YAML file entirely instead of disabling it. In this case, explain the removal in the commit message.
- **Documenting Constraints**: If you disable an option for a specific architecture (e.g., `-riscv`) or configuration (e.g., `-kmsan`), or if you make some config 'weak', add a comment explaining this decision.
- **Explicit Constraints**: Prefer explicitly enumerating the supported platforms and/or kernel versions instead of using `[optional]` or `[weak]` constraints when possible. If the option was added in `linux-next`, use the tag `[linux-next]`. If the specific kernel version is not known but it is in a released kernel, use `[weak]` and add a comment explaining it. Also use `[weak]` or `[optional]` if the config option has dependencies that might not be met in all configurations (e.g., depends on specific networking features or hardware support). **Do not silently make a failing config `weak` without investigation. Prefer identifying the specific reason (e.g., architecture limitation or kernel version change) and using explicit exclusions if possible.**
- **File-Level Exclusion**: If all options in a configuration fragment are disabled for certain kernel versions or instances, prefer excluding the whole file in `main.yml` using negative tags (e.g., `-upstream`) instead of adding version constraints to every individual option in the fragment. Add an explanatory comment above the inclusion line in `main.yml` to explain why it is disabled.
- **Explanation Headers**: Every new file created in `dashboard/config/linux/bits` requires an explanation header describing the purpose of the file, why we need it, and how it is used.
- **No Manual Edits**: Do NOT manually update the generated `.config` files. They MUST always be generated by running `make configs`.

## Prerequisites
- A local clone of the `linux` kernel repository.
- A local clone of the `syzkaller` repository.
- Docker installed for running `syz-env`.

## Procedure

### 1. Update Kernel Commits
Modify the corresponding YAML files in `dashboard/config/linux/bits/` to point to the desired tags. Use tags instead of commit hashes when updating these files.
- `linux-upstream.yml`
- `linux-next.yml`

### 2. Run Config Generation
Execute the following command from the root of the `syzkaller` repository:
```bash
CI=true ./tools/syz-env make configs SOURCEDIR=/path/to/linux/source
```
*Note: Replace `/path/to/linux/source` with the absolute path to your local Linux kernel directory.*

### 3. Troubleshooting Common Issues

#### A. Kconfig Parser Limitations (`syz-kconf`)
The `syz-kconf` parser in `pkg/kconfig` may not support advanced Kconfig syntax such as `if` in `depends on` statements.
- **Solution:** Implement support for the missing syntax in `pkg/kconfig/parser.go`. Do NOT modify Kconfig files in the kernel source.

#### B. Missing Config Options in Final Config
If an option specified in a YAML fragment is not present in the generated `.config`, `make configs` will fail.
- **Check Dependencies:** Verify if the option depends on another option not enabled, or if it is not supported on the target architecture or kernel version.
- **Version Constraints:** Add version constraints to the option in the YAML file, e.g., `[-v7.1]` to disable it for versions starting from `v7.1`.
- **Weak Constraints:** If the option is optional and may not be present in all versions, make it weak: `[value, weak]`. This prevents failure if the option is missing. **Avoid using `weak` just to bypass errors. Investigate why the option is missing or failing to be set. If it is due to architecture or version specific behavior, use explicit exclusions (e.g. `[-arm]`) instead of `weak`.**

#### C. Git Fetch Failures
If `syz-kconf` fails to checkout a repository or commit (e.g., Android kernels):
- **Diagnosis:** Check the error message to find the auto-generated remote name (often a hash).
- **Workaround:** Manually run `git fetch <remote_name>` in the kernel source directory to pull all objects, then retry `make configs`.

### 4. Finalizing Changes
Once `make configs` completes successfully:
1. Stage and commit the changes to YAML files first.
2. Stage and commit the generated `.config` files as the VERY LAST commit in the PR. The commit message MUST be `dashboard/config/linux: update generated configs`.
3. Push to a separate branch and create a Pull Request.

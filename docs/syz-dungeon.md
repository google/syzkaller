# Syzkaller Dungeon

## Overview
The **Syzkaller Dungeon** (/syz-dungeon) ranks contributors (kernel developers) based on their achievements in fixing bugs found by Syzkaller. It uses a retro "terminal hacker" aesthetic with green-on-black text and ASCII art.

Note: Currently, the Dungeon is only available for the `upstream` namespace.

## Core Mechanics

### 1. Character Generation
Each contributor is assigned a "Character" based on their bug fixing history during a specific time period.

**Eras (Time Ranges)**
The Dungeon provides two separate views or "Eras" for statistics:
*   **1 Year (Default)**: A rolling window of exactly the last 365 days from today.
*   **All Time**: A comprehensive view of all bugs fixed since the beginning of tracking.

*   **Era Scoping & Decay**: In the 1-Year Era, the **entire character sheet** (including XP, Level, Class, Badges, and Stats like STR/INT/WIS) is dynamically recalculated based exclusively on the bugs currently within the 365-day rolling window. As fixes age out of the window, these attributes decay. In the All-Time Era, bugs never fall out of the window, providing a permanent, non-decaying global view of the contributor's achievements.
*   **Eligibility**: Bugs must have landed a fix commit within the given period.
    *   **Criteria**: Includes both fully fixed bugs (on the **Fixed** page) and partially fixed bugs (still **Open** but with upstream commits landing in the given period).
    *   **Co-Authored Fixes**: If a bug is resolved by multiple commits, **all unique commit authors** recorded in the bug's history are independently granted 1 bug count and the associated XP/Score. This means a single bug can boost multiple heroes simultaneously, occasionally resulting in leaderboard totals that are higher than naive SQL queries that only map one author per bug.
    *   **Date**: The "date fixed" is determined strictly by `FixTime` (when the fix commit was discovered). `Closed` date is ignored.
*   **Matching**: Identity resolution uses **strict exact matching** on the lowercase **Email** address (`Author` field).
    *   **No Email Grouping**: Contributors using different email addresses (e.g., `@gmail.com` vs a corporate domain) are treated as separate characters.
*   **Hero ID**: Each hero is assigned a stable, 16-character hexadecimal ID. This is generated by calculating the hash of their **lowercased email address** (using SHA-1) and extracting the first 16 characters of the hexadecimal string representation of the hash. This provides a secure, deterministic identifier for individual Hero Profile Pages without exposing raw email addresses in URLs.
*   **Hero Name**: Derived from the `AuthorName` field in the Datastore `CommitInfo` entity, bound to the matched Email.
        *   If a contributor uses multiple names for the same email (e.g. `John Doe` vs `J. Doe`), the system deterministically assigns the **most frequently used name** to the Hero. Ties are broken alphabetically.
    *   **Fallback**: If an email *never* encounters a valid author name across all of its recorded bugs, the hero's name will gracefully fall back to displaying just the raw email address.
*   **Bots**: Bots are implicitly excluded as only humans land fixing commits in the Linux kernel (even bot-generated patches are signed by a human).
*   **Heroic Title**: An adjective is appended to the name based on the first letter, e.g., "Eric Dumazet The Epic".
    *   **Full List**:
        *   `A`: The Arcane
        *   `B`: The Brave
        *   `C`: The Cosmic
        *   `D`: The Devoted
        *   `E`: The Epic
        *   `F`: The Fierce
        *   `G`: The Gallant
        *   `H`: The Heroic
        *   `I`: The Immortal
        *   `J`: The Just
        *   `K`: The Knightly
        *   `L`: The Legendary
        *   `M`: The Mystic
        *   `N`: The Noble
        *   `O`: The Omniscient
        *   `P`: The Prime
        *   `Q`: The Quick
        *   `R`: The Radiant
        *   `S`: The Sacred
        *   `T`: The Titan
        *   `U`: The Undying
        *   `V`: The Valiant
        *   `W`: The Wise
        *   `X`: The Xenial
        *   `Y`: The Youthful
        *   `Z`: The Zealous
    *   *Identity:* Defined by the dynamic **Hero Name** calculated above (which selects the first valid name encountered for the email, or falls back to the raw email).
        *   **No Title for Fallbacks**: If the Hero Name had to fall back to the raw Email because no valid name was found across their commits, **no heroic title modifier is added**. The hero is simply displayed as their address (e.g. `raw.email@domain.com`).
    *   **Logic**: The title is determined by the **first letter of the resolved name**.
    *   **Normalization**: The first letter is **lowercased** before lookup.
    *   **Map**: 26 adjectives corresponding to 'a'-'z'.
    *   **Fallback**: If the name begins with a non-alphabetic character (numbers, symbols, or non-Latin scripts not in the a-z map), the title defaults to **"The Adventurer"**. This fallback applies universally without transliteration (no UTF-8 awareness).
    *   *Note:* There is currently no mechanism for manual identity overrides or merging separate profiles.
*   **Class**: Determined by analyzing the **existing tags** attached to the bug.
    *   **Multi-Class Credit**: A contributor receives exactly **1 point** for **every** subsystem tag present on a fixed bug. For example, a bug tagged with `net` and `mm` increments the contributor's count for both `net` and `mm` by 1.
    *   **Dynamic Class**: The "Main Class" is determined **after** calculating the totals for all valid bugs. The system finds the single subsystem tag that appears most frequently across all of the contributor's fixes, and assigns the Class corresponding to that winning tag.
    *   **Source**: Classes are determined by mapping the winning **tag** (subsystem) to the RPG categories below.
    *   **Weaver** (🕸️): Networking & BPF.
        *   Tags: `net`, `ipv4`, `ipv6`, `tcp`, `udp`, `sctp`, `bpf`, `wireless`, `bluetooth`, `can`, `tipc`, `nfc`, `bridge`, `netlink`.
    *   **Geomancer** (🏔️): Filesystems & Storage.
        *   Tags: `fs`, `ext4`, `btrfs`, `xfs`, `vfs`, `block`, `scsi`, `nvme`, `io-uring`, `fuse`, `overlayfs`.
    *   **Voidwalker** (🔮): Memory Management.
        *   Tags: `mm`, `kasan`, `slab`, `kcsan`, `kmemleak`.
    *   **Paladin** (🛡️): Security & Crypto.
        *   Tags: `security`, `selinux`, `apparmor`, `lsm`, `audit`, `keyrings`, `crypto`.
    *   **Bard** (🎻): Media, Sound, Input, Human Interface.
        *   Tags: `sound`, `usb`, `media`, `video`, `drm`, `gpu`, `input`, `hid`, `drivers` (generic).
    *   **Artificer** (🛠️): Hardware Buses & Infrastructure.
        *   Tags: `pci`, `i2c`, `spi`, `gpio`, `clk`, `pm`, `dmaengine`, `iio`, `rtc`, `watchdog`.
    *   **Warlock** (🌀): Virtualization.
        *   Tags: `kvm`, `xen`, `virt`, `hyperv`.
    *   **Rogue** (🗡️): Architecture Specific.
        *   Tags: `arch`, `arm`, `arm64`, `x86`, `riscv`, `s390`.
    *   **Druid** (🌿): Core Kernel & Sync.
        *   Tags: `kernel`, `sched`, `rcu`, `locking`, `trace`, `cgroups`.
    *   **Warrior** (⚔️): Default class for other subsystems or if **no tags** match any class.
    *   *Tie-Breaking:* If a contributor has multiple tags with the exact same highest frequency count, the **first subsystem tag alphabetically** is chosen as the winner. Warrior is only used if there are no specialized tags.

### 2. Experience (XP) Calculation
Score is calculated for each fixed bug:
*   **Base XP**: 100
*   **Boss Bonus**: 10% of crash count (Capped at 500 XP).
    *   Formula: `ceil(NumCrashes * 0.1)`
    *   *Note:* If the bug data is missing crash counts or has 0 crashes, this bonus is 0.
*   **Ancient Evil Bonus**: 1 XP for every day (measured as a precise 24-hour elapsed duration, i.e., `hours / 24`) the bug was open (Capped at 300 XP).
    *   *Note:* if `FirstTime` is missing, this bonus defaults to 0.
    *   *Floor:* If duration is negative (due to clock skew), it is treated as **0 days** (0 bonus XP). This prevents "negative XP".
*   **Windwalker Bonus**: +250 XP flat bonus if the bug was fixed within 7 days. This bonus defaults to 0 if either the `FirstTime` or `FixTime` timestamps are missing.
*   **Blind Bonus**: 150 XP if the bug had no reproducer (`ReproLevelNone`).

### 3. Leveling
*   **Level 1**: 0 XP
*   **Progressive Level Base**: 1,000 XP per level up to Level 50.
*   **Legendary Scaling**: Starting at Level 50, each subsequent level requires 500 *more* XP than the previous level's requirement (e.g., Level 50 -> 51 costs 1,500 XP, 51 -> 52 costs 2,000 XP).
*   **Mathematical Formula**: The total XP required to reach Level $N$, denoted as $TotalXP(N)$, is calculated strictly by the following piecewise function:
    *   For $N \le 50$: $TotalXP(N) = (N - 1) \times 1000$
    *   For $N > 50$: Let $m = N - 50$. The total XP is defined by the arithmetic series sum:
        $TotalXP(N) = 49000 + 1000m + 250m(m + 1)$
*   **Concrete Threshold Table**:
    *   `Level 1`: 0 XP
    *   `Level 2`: 1,000 XP
    *   `Level 49`: 48,000 XP
    *   `Level 50`: 49,000 XP
    *   `Level 51`: 50,500 XP
    *   `Level 52`: 52,500 XP
    *   `Level 53`: 55,000 XP
*   **Cap**: There is no hard level cap.
*   **Note**: Rapid leveling (e.g., jumping 2.5 levels from a single legendary bug fix) is **intentional** to reward high-impact contributions.

### 4. Inventory (Badges)
Contributors earn badges for specific achievements. **Badges are purely cosmetic** and do not affect XP or Rank.

**General Badge Rules:**
*   All string/title matching is **case-insensitive**.
*   Bugs missing `FirstTime` or `FixTime` fields cannot be evaluated and therefore do not count toward time-based badges.
*   **Dragon Vanquisher** (🐉): Fixed a bug with >1000 crashes.
*   **Necromancer** (💀): Fixed a bug open for >365 days.
*   **Diviner** (👁️): Fixed a bug with no reproducer.
*   **Windwalker** (🪽): Fixed a bug within **7 days** (measured as precise 24-hour duration chunks) of it being reported (based on `FirstTime`).
    *   *Logic:* `FixTime - FirstTime < 7 days`.
    *   *Data Integrity:* This calculation is **skipped** (badge not awarded) if either `FirstTime` or `FixTime` timestamps are missing.
*   **🔥 Hydra Hunter**: Fixed 10 or more bugs overall.
*   **🧌 Troll Crusher**: Fixed 50 or more bugs overall.
*   **🐙 Kraken Bane**: Fixed 100 or more bugs overall.
*   **👹 Demon Slayer**: Fixed 250 or more bugs overall.
*   **🔱 Leviathan Conqueror**: Fixed 500 or more bugs overall.
*   **🏺 Sealer**: Fixed a memory leak.
    *   *Logic:* Title or Fix Commit Title contains tokens `leak`, `leaks`, `memory-leak`, or `memory-leaks` (with word boundaries checking).
*   **📜 Keeper of Secrets**: Fixed an *infoleak* vulnerability.
    *   *Logic:* Title or Fix Commit Title contains `infoleak` (matched as a simple substring, without word boundaries or special character tokenization).
*   **⏳ Master of Time**: Fixed a time-related bug.
    *   *Logic:* Title or Fix Commit Title contains `time:` or `timer:` (matched as a simple substring, without word boundaries or special character tokenization, matching things like `hrtimer:` or `posix-timer:`).
*   **🛶 Pathfinder**: Fixed an *out-of-bounds* error. (String match: `out-of-bounds` anywhere in bug title/commit titles).
*   **🦄 Beast Tamer**: Fixed a *data race* condition. (String match: `data-race` anywhere in bug title/commit titles).
*   **⚒️ Armorer**: Fixed a *general protection fault*. (String match: `general protection fault` anywhere in bug title/commit titles).
*   **🪦 Grave Robber**: Fixed a *use-after-free* error. (String match: `use-after-free` anywhere in bug title/commit titles).
*   **🗝️ Locksmith**: Fixed a *locking* issue. (Matches tokens: `lock`, `deadlock`, `lockup`, `locking`, `mutex`, `unlock`, `locks`, `rwlock`, `spinlock` in bug title/commit titles).
*   **🕯️ Exorcist**: Fixed a *hung task* or *zombie process*. (String match: `hung task`, `task hung`, or `zombie` anywhere in bug title/commit titles).
*   **🕳️ Abyss Walker**: Fixed a *NULL pointer dereference*. (String match: `null-ptr-deref` or `null pointer` anywhere in bug title/commit titles).
*   **⚗️ Alchemist**: Fixed an *uninitialized value* bug. (String match: `uninit` or `kmsan` anywhere in bug title/commit titles).
*   **🪓 Executioner**: Fixed a *double free* bug. (String match: `double-free` or `double free` anywhere in bug title/commit titles).

### 5. General Rules

*   **Alphabetical Tie-Breakers**: Unless otherwise specified, all "alphabetical" sorting and tie-breaking used throughout the Dungeon (for Names, Kingdom Main Classes, Guilds Array, Kingdom Ranks, etc.) is strictly **case-sensitive** and **lexicographically ascending**. This guarantees absolute determinism derived from underlying ASCII/UTF-8 byte representations (e.g., uppercase 'Z' sorts before lowercase 'a').

### 6. Kingdoms

Players can view aggregate statistics via two toggles in the dashboard menu:

*   **Entities**: Choose between viewing the standard individual **Heroes** leaderboard, or the **Kingdoms** ranking.
*   **Kingdoms**: View aggregated points and heroic classes across different organizations playing the dungeon.
    *   Clicking a Kingdom's name takes you to the **Kingdom Profile Page** where you can see their overall Trophies, Hero Count, Total XP, and established Guilds.
    *   Employees are assigned to their company based on a curated list of email domains. The matching rule requires an **exact string match** against the entire portion of the email address following the `@` symbol (e.g., `developer@corp.company.com` will *not* match `company.com`).
        *   **The Google Kingdom**: `google.com`, `chromium.org`
        *   **The Red Hat Kingdom**: `redhat.com`
        *   **The Huawei Kingdom**: `huawei.com`, `hisilicon.com`
        *   **The Linux Foundation Kingdom**: `linuxfoundation.org`, `kernel.org`
        *   **The Meta Kingdom**: `meta.com`, `fb.com`
        *   **The Intel Kingdom**: `intel.com`, `linux.intel.com`
        *   **The AMD Kingdom**: `amd.com`
        *   **The ARM Kingdom**: `arm.com`
        *   **The Oracle Kingdom**: `oracle.com`
        *   **The SUSE Kingdom**: `suse.com`, `suse.de`, `suse.cz`
        *   **The Linaro Kingdom**: `linaro.org`
        *   **The Alibaba Kingdom**: `linux.alibaba.com`
        *   **The NVIDIA Kingdom**: `nvidia.com`, `mellanox.com`
        *   **The IBM Kingdom**: `linux.ibm.com`, `linux.vnet.ibm.com`
        *   **The Bytedance Kingdom**: `bytedance.com`
        *   **The Canonical Kingdom**: `canonical.com`, `ubuntu.com`
        *   **The Amazon Kingdom**: `amazon.com`, `amazon.de`, `amazon.co.uk`, `amazon.co.jp`
        *   **The Tencent Kingdom**: `tencent.com`
        *   **The Samsung Kingdom**: `samsung.com`
    *   Any hero not in the curated list belongs to `The Independent Mercenaries`.
    *   **Royal Champion**: A Kingdom's Royal Champion is determined by finding the single hero associated with that Kingdom who has the highest `Score` (XP) contribution in the given Era. The Champion's Name, XP contribution, and a link to their profile are featured on the Kingdom's Profile Page.
    *   **Total Hero XP**: A Kingdom's total `Score` is purely the combined sum of the `Score` of all its affiliated `Heroes`. This means all dynamic bug bonuses (like the Boss Bonus for crashes, Ancient Evil bonus for old bugs, etc.) earned by a hero directly benefit their aligned Kingdom. However, the Kingdom's hero count and specific class composition metric will correctly reflect all distinct individuals who participated. If the fix is co-authored by heroes from **different** Kingdoms, **both** Kingdoms independently receive full credit for the bug.
    *   **Trophy Deduplication**: To prevent double-counting, if a bug is resolved jointly by multiple heroes who belong to the **same** Kingdom, the Kingdom is credited exactly **1 Trophy** for resolving the bug.
    *   **Stats**: Kingdoms do not possess individual stats like STR, INT, or WIS.
    *   Instead of "Inventory" (Badges), Kingdoms display **"Guilds"**, which details the composition of Hero Classes based on the **total count of players** in the Kingdom (e.g., "3 Weavers", "1 Geomancer"). The array is sorted descending by the count of players, and then alphabetically by the resulting string representation in the event of a tie. Pluralization is handled simply: an `"s"` is appended to the Class Name if the player count is strictly greater than 1.
    *   **Sorting**: Kingdoms are ranked and sorted descending by their **Total XP** (Score). Ties are broken alphabetically by the Kingdom Name.
    *   **Territory Tiers**: A Kingdom evaluates its overall combined `Score` (XP) strictly into one of the following brackets, displaying an associated name and emoji. On the **Kingdom Profile** page, the tier icon is displayed prominently with the tier name below and an XP range tooltip. On the global **Leaderboard**, the tier emoji is displayed as an inline badge next to the Kingdom's name (with a tooltip revealing the tier name).
        *   0 - 49,999: Outpost (⛺)
        *   50,000 - 149,999: Barony (🏘️)
        *   150,000 - 499,999: Duchy (🏛️)
        *   500,000 - 999,999: Kingdom (🏰)
        *   1,000,000+: Empire (👑)


## Technical Implementation

### Backend (`dashboard/app/dungeon.go`)
*   **JSON API**: `/syz-dungeon?json=1` returns the **calculated scaled stats** (RPG range [3-30]) for `str`, `int`, `wis`.
*   **Endpoint**: `/syz-dungeon` (Leaderboard Overview)
*   **Hero Profile Endpoint**: `/syz-dungeon/hero/{id}` (Individual Hero Sheet)
    *   **Parameters**: Supports `?json=1` (JSON dump) and `?era=1y` or `?era=all` (defaults to `all`). Hero pages will display an interactive `[1 YEAR]` | `[ALL TIME]` toggle if the hero has contributions mapped into both era scopes.
    *   **Class Box**: Displays the hero's class emoji, name, and a tooltip containing the class's flavor text (description).
    *   **Attributes Box**: Displays the hero's STR, INT, and WIS properties. Hovering over each stat reveals a tooltip explaining its corresponding metric (e.g., "Max Crashes (Scaled)").
    *   **Badges Box**: Displays earned badges. Hovering over a badge reveals its name and an extended description characterizing the achievement.
    *   **Click-to-Copy ID**: The `ID` displayed in the footer is clickable and will copy the URL to the profile into the user's clipboard.
* **Data Source**: Queries the `Bug` entity in Google Cloud Datastore.
    * **Definition**: `dashboard/app/entities_datastore.go`
    * **Key Fields Used**:
        * `Namespace`, `Status`: For Datastore query filtering. (Note: Only the `upstream` namespace is currently supported).
        * *Logic:* `Status` must be Fixed (or Open if fixing commits are present).
        * *Index:* Requires composite index on `(Namespace, Status)` in `index.yaml`.
        * `FixTime`: To filter bugs within the given period (in **UTC**).
            *   *Decay:* Since the cache refreshes hourly, players viewing the 1-Year Era will see their entire character sheet—including stats, badges, levels, classes, and ranks—actively decay day-to-day as old bugs fall out of the 365-day window. **No notifications** are sent for level down; users monitor their status on the dashboard.
        * `Tags`: For Class determination (instead of Title).
        * `CommitInfo`: For authorship identification (`AuthorName` and `Author` fields).
        * `Title`: For "Sealer", "Keeper of Secrets", and "Master of Time" badge logic.
        * `NumCrashes`: For "Boss Bonus", "Dragon Slayer" badge, and **STR**.
        * `ReproLevel`: For "Blind Bonus", "Diviner" badge, and **INT**.
        * `FirstTime` & `FixTime`: For "Ancient Evil" bonus, "Windwalker"/"Necromancer" badges, and **WIS**.
* **Caching**: Results are cached in shared **App Engine Memcache** (using compressed JSON) for 1 hour. The cache is segregated by the viewer's `AccessLevel` so that privileged bug data (like undisclosed bugs visible to admins) does not leak to public viewers.
    *   *Warm-up/Preheating*: A cron job (`/cron/dungeon_preheat`) runs hourly to precompute and warm the cache for all access levels simultaneously, ensuring fast page loads for all users across multiple live instances.
* **Logic**:
    * **Identity Resolution**: Aggregates bugs strictly by the lowercased **Email** address.
        * *Rule:* "Linus Torvalds <email1>" and "Linus Torvalds <email2>" are treated as separate characters.
        * *Name Selection*: If an author uses multiple names for the same email address across different commits, the system assigns the **most frequently used name** to the Hero. Ties are broken alphabetically.
    * Determines the "Main Spec" (class) based on subsystem counts.
    * Calculates XP and assigns badges.
    * Sorts players by Score (descending), then Name (ascending), then Email (ascending).
    * **Pagination**: The backend server returns the complete active roster of players. The frontend JavaScript dynamically partitions the tables into interactive pages of 100 rows.

### Frontend (`dashboard/app/templates/dungeon.html`)
*   **Style**: Dark mode with `Courier New` / monospace font.
    *   Accent Color: Neon Green (`#0f0`).
    *   Banner: Retro pixel-art PNG image "SYZKALLER DUNGEON" (responsive, located at `/static/dungeon.png`).
* **Interactivity**:
    * **Hover Tooltips**: Hovering over a character row reveals a "Character Sheet":
        * **Desktop Users**: Appears on **Hover**.
        * **Mobile/Touch Users**: Appears on **Click/Tap**. (Tapping a row toggles the sheet; opening a new one closes the previous one).
        * **NAME**: "Name Adjective" (e.g., "Eric Dumazet The Epic")
        * **CLASS**: "Emoji (Class)" (e.g., "🕸️ (Weaver)")
        * **LEVEL**: Current Level
      *   **Stats Scaling**:
    *   Instead of raw counts, metrics are mapped to an RPG-like scale **[3, 30]** using a logarithmic function.
    *   **STR** (Strength): Based on **Max Crashes** of a single fixed bug.
        *   Formula: $STR = \min(30, \max(3, \lfloor 3 + 4 \times \log_{10}(\max(1, \text{MaxCrashes})) \rfloor))$
        *   *Example:* 1 crash = 3, 100 crashes = 11, 1,000,000 crashes = 27.
    *   **INT** (Intelligence): Based on total **Blind Fixes** (no reproducer).
        *   Formula: $INT = \min(30, \max(3, \lfloor 3 + 10 \times \log_{10}(\text{BlindFixes} + 1) \rfloor))$
        *   *Example:* 0 blind fixes = 3, 2 blind fixes = 7, 10 blind fixes = 13, 500 blind fixes = 30.
    *   **WIS** (Wisdom): Based on **Oldest Kill** (maximum duration of a **single** bug).
        *   Formula: $WIS = \min(30, \max(3, \lfloor 3 + 5 \times \log_{10}(\text{DaysOpen} + 1) \rfloor))$
        *   *Example:* 1 day = 4, 100 days = 13, 1000 days = 18, 250,000 days = 30.
    * **Inventory Tooltips**: Hovering over badges shows their name/description.
